# BLD-001 — Steer UDP/5000 to RX queue 0 with DOCA Flow

**Skill primarily exercised.** `skills/libs/doca-flow/` (with `doca-version`
and `doca-debug` overlays).

## Prompt (run identically against every config)

```
I have a BlueField-3 with DOCA installed. I want to write a tiny C
program that uses DOCA Flow to match every UDP packet whose destination
port is 5000 and steer it to RX queue 0 of port representor 0. I want
the smallest end-to-end working example I can build with pkg-config —
not a full pipeline scaffold, just one match + one action.

Walk me through:

  (1) How do I confirm DOCA Flow is installed at a compatible version,
      and which symbol do I call to confirm the device supports the kind
      of action I'm about to use, BEFORE I commit to writing the
      pipeline?
  (2) Start from a real shipped DOCA Flow sample under
      /opt/mellanox/doca/samples/doca_flow/. Name the sample, the files
      I will edit, and the minimum-change set.
  (3) Build it with `pkg-config --cflags --libs doca-flow` (not
      hand-crafted -l flags), run it once, and name the concrete
      observable that proves UDP/5000 is being steered to RX queue 0.
  (4) If the rule appears installed but the counter on RX queue 0 does
      not increment, what's the FIRST debug step I take — and at which
      of the 7 debug layers?
```

## Expected bundle-sourced symbols / paths / commands

A passing answer cites at least the following (the bundle's deep-E2E
grader rewards any *concrete* citation; this list is the minimum):

- `pkg-config --modversion doca-common` and `doca_caps --version` as
  step-1 version probes (from `skills/doca-version/CAPABILITIES.md
  ## Version compatibility`).
- The actual cap-query symbol for DOCA Flow (from
  `skills/libs/doca-flow/CAPABILITIES.md ## Capabilities and modes`) —
  not invented from a template.
- A real sample name under `/opt/mellanox/doca/samples/doca_flow/` (e.g.
  `flow_port_fwd`, `flow_switch_single`) named verbatim from
  `skills/libs/doca-flow/TASKS.md ## build`.
- `pkg-config --cflags --libs doca-flow` exactly — no synthesized `-ldoca_flow`.
- A named observable: `doca_flow_query_entry` / `doca_flow_query_pipe_miss`
  / `doca_flow_pipe_dump` (from `skills/libs/doca-flow/TASKS.md
  ## test`).
- For step 4: layer-5 (Runtime) per the 7-layer ladder in
  `skills/doca-debug/CAPABILITIES.md`, with the read-only triple
  (`doca_flow_pipe_dump` + RX queue counter + `--sdk-log-level 70` SDK
  log) as the first debug move.

## Failure modes the grader explicitly catches

| Failure | Dimension that flags it |
|---|---|
| Names a `doca_flow_cap_*` symbol that does not exist in `libs/doca_flow/include/public/*.h` | D1_invented_tokens — FAIL |
| Skips step 1 (jumps straight to writing code) | D2_sequence_correctness — FAIL |
| Says *"check the logs"* without naming a counter / dump symbol | D3_validation_concreteness — FAIL |
| Says *"run dmesg"* without identifying the debug layer first | D4_debug_concreteness — FAIL |
| Hands the user pseudocode (no real pkg-config + meson invocation) | D5_consumer_concreteness — FAIL |
